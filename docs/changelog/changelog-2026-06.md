# CHANGELOG JUNE 2026

## June 2nd, 2026 - feat(lists): rewire all mutators and readers to object items (items-as-objects, slice 3 of M8 foundation)

The muscle slice: every List mutator and server-side reader now speaks the object shape, `ListItemTimestamps` is deleted, and the `item_added_at` column is dropped. After this, the backend is fully on objects; only the Vue client mirror (overlay renderer + dashboard live-update handler) remains, which is the next slice.

- **Mutators rewired through `ListItems`**: chat append (`ListAppendService`) and control-driven append (`ListWriterAppend`, incl. FIFO) allocate an id from `next_item_id` and append an object; draw/pop/clear (`ListActionService`) remove by index via `ListItems::removeAt` and resolve the winner/popped value off the object; clone copies the item objects verbatim and inherits `next_item_id` so a later append can't reuse a copied id; dashboard create uses `ListItems::freshFromValues`, dashboard save uses `ListItems::reconcileByValue` (preserves a matched item's id/age/label/weight/color across a reorder); snapshot restore uses the new `ListItems::adopt` (keeps ids + rich fields, refreshes `added_at` so an old snapshot isn't instantly TTL-swept, reseats `next_item_id`); the entry-TTL sweeper reads each item's own `added_at`; the recipe installer wraps manifest string arrays into objects at install (manifest schema unchanged).
- **Readers made object-aware**: `ListActionService` count/first/last/random/search/searchall read values via `ListItems::values`; `Picker::fire()` (the recipe draw engine) resolves the picked item's value; the `OverlayTemplateController` `c:list:*` tag projection (`.N`, `:first`, `:last`, `:random`, `:sum`, the bare JSON-array tag) projects to value strings for full backward compatibility (rich `.N.label` access is a later additive slice); the dashboard list payload (`ListController::serialize`) and snapshot-listing endpoint expose value strings so the Vue editor and content-search keep working untouched this slice.
- **`next_item_id`** added to `OptionSet` fillable/casts; `items` PHPDoc now documents the object shape. `ListItemTimestamps` deleted. New migration `2026_06_02_130000_drop_item_added_at_from_option_sets` drops the now-unreferenced parallel column (down() rebuilds it from each item's `added_at`).
- **`ListItems::adopt()`** added with unit coverage: re-ingests existing item objects (snapshot restore), preserving ids and rich fields, optionally refreshing `added_at`, and reseating the id counter past the highest restored id.
- Tests: every List/Recipe fixture now builds object items; value assertions go through `ListItems::values`; timestamp-sensitive tests assert each item's `added_at`; the `ListUpdated` event payload (now object items) asserts via `values`. The two pure `ListItemTimestamps` unit-tests in `ListExpiryTest` were removed (superseded by `tests/Unit/ListItemsTest.php`). Full List/Recipe suite green (215 passed); whole suite green apart from one pre-existing, unrelated Freesound-cap failure.

## June 2nd, 2026 - feat(lists): convert List items to objects (items-as-objects, slice 2 of M8 foundation)

Schema migration turning `option_sets.items` (and `list_snapshots.items`) from flat string arrays into the rich object shape from slice 1, and adding the per-list `next_item_id` counter. `["A","B"]` + `item_added_at:[t1,t2]` becomes `[{id:1,value:"A",added_at:t1,label:null,weight:1,color:null}, ...]`. The per-item timestamp now lives inside the item, which is what lets the fragile parallel `item_added_at` array (and its sync discipline) be retired in a later slice. Rehearsed up-and-down on real local Postgres data before committing, per the DB-change discipline; production is untouched (slice 5, on explicit go only).

- **`database/migrations/2026_06_02_120000_convert_option_set_items_to_objects.php`**: converts both tables via `ListItems::fromLegacy`, ids assigned 1..n per list, `next_item_id` set to count+1. A missing or empty `item_added_at` (recipe-installed lists that predate the TTL column) falls back to the row's `created_at` - so those rows are healed, not lost. Snapshots never tracked per-item stamps, so restored items age from the snapshot's own `created_at`. Both directions are idempotent (guard on the current shape), so a re-run or half-applied state is safe.
- **Deliberately does NOT drop `item_added_at`**. The mutators, readers, and tests that still reference it are rewired in slices 3-4, and a `RefreshDatabase` test run executes this migration - dropping the column here would break the still-string-era suite. A small follow-up migration drops it at the end of slice 3, once nothing references it. Until then the column lingers, stale and ignored; the objects' `added_at` is the source of truth.
- Writes go through `DB::table` with hand-rolled `json_encode` rather than the Eloquent model, on purpose: a later slice removes `item_added_at` from the model casts, and a model-based write would then mishandle the very columns this migration converts. Confirmed the `DB::table` jsonb write works cleanly on Postgres (no text-to-jsonb cast error).
- Verification on real data: lists with matched stamps preserved their timestamps exactly; lists with empty stamps got `created_at`-derived stamps and sequential ids; the down migration restored the original string arrays and rebuilt `item_added_at`. Full List suite green (one unrelated pre-existing second-boundary timestamp flake in `ListExpiryTest` that passes on retry, and which slice 3 removes anyway).

## June 1st, 2026 - feat(lists): add ListItems chokepoint helper (items-as-objects, slice 1 of M8 foundation)

First slice of the items-as-objects work that turns List entries from plain strings (`["a","b"]` plus a parallel `item_added_at` array) into rich objects (`{id, value, added_at, label, weight, color}`). This slice is purely additive scaffolding: a single, pure, fully-tested helper class that every mutator and the upcoming schema migration will funnel through. Nothing calls it yet, so behavior is byte-identical and there is zero production risk - it is the foundation the rest of M8 stands on.

- **`app/Support/ListItems.php`**: the one place List item objects are built and validated. `make()` is the single validating constructor - value coerced to string (empty preserved, never stripped), weight clamped to a positive number defaulting to 1, color kept only if it is a valid `#rgb`/`#rrggbb` hex (else null), label kept only if a non-empty string (else null). Builders on top of it map 1:1 onto the real call sites: `appendValue()` (chat + control-driven append), `removeAt()` (draw + pop), `freshFromValues()` (store / restore / recipe install), `reconcileByValue()` (dashboard textarea save - preserves a matched item's id/age/color/weight across a reorder, mints fresh items for new values, oldest-match-wins on duplicates), `fromLegacy()` (the schema migration's string+timestamp -> object converter), and `values()` (the back-compat anchor every scalar reader leans on). The per-item `added_at` now lives inside the item, which is what lets the fragile parallel `item_added_at` array and its `ListItemTimestamps` sync discipline go away in a later slice.
- **`tests/Unit/ListItemsTest.php`**: 23 tests / 57 assertions covering the full surface - validation edges (weight clamping, color validation + lowercasing, empty-value preservation, label absence), id allocation and `next_id` reporting, the reorder/duplicate/removal semantics of `reconcileByValue`, and the legacy-conversion fallbacks. All green; Pint clean.
