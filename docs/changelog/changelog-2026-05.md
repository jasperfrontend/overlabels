# CHANGELOG MAY 2026

## May 12th, 2026 - Per-alert TTS playback delay so the alert sound can play before the voice

- The first cut of TTS spoke the rendered string the moment the broadcast arrived, which collided with the alert sound/animation every streamer's templates already include. Standard alert design is "noise first, voice second" - and as long as we don't give people a way to push the voice back, every alert is going to sound like two things shouting at once. Fix: a per-alert-template delay value, configurable on the same TTS tab that already holds the expression.
- New `tts_delay_ms` column on `overlay_templates` (unsigned int, default 0, capped at 60000 by the controller validator). Sits next to `tts_expression` because they're authored together and conceptually inseparable - a delay without an expression is meaningless, and an expression without a delay is just the v1 behaviour we already had. Cast to integer on the model so `null` from the DB never sneaks into the JS layer as a string.
- `AlertTriggered` gained an `int $ttsDelayMs = 0` constructor param (clamped to `max(0, value)` in the constructor so a freak negative can never reach the wire) and a `tts_delay_ms` key in `broadcastWith()`. All 3 dispatch sites - `TwitchEventSubController::renderEventAlert`, `ExternalAlertService::dispatch`, `ExternalEventController::replay` - pass `$template->tts_delay_ms ?? 0`. The constructor param defaults to 0 so existing test fixtures that don't care about it keep compiling.
- Overlay side: `OverlayRenderer::speakTts(text, delayMs)` now schedules speech via `setTimeout` when `delayMs > 0`. The pending timer is held in a module-scoped `ttsPendingTimer` ref so a new alert arriving during the delay window cancels the previous alert's not-yet-spoken voice - otherwise a fast double-fire would queue up two delayed `speak()` calls with no way to interrupt the first. `clearTimeout` on the pending timer + `speechSynthesis.cancel()` runs unconditionally before scheduling, which collapses cancel-pending + cancel-in-flight + schedule-new into one path.
- Editor UI: a number input "Wait before speaking (ms)" sits below the TTS Expression textarea on the same tab. Step is 100, min 0, max 60000, body copy explains the "match the length of your alert sound" intent. Form binding uses `v-model.number` so the value reaches the backend as an integer, not a string.
- 4 new feature tests: ships the value through `ExternalEventController::replay`, defaults to 0 when the column is absent, exposes the field through `broadcastWith()`, and the negative-input clamp. 545 tests green (was 540).
- Deliberately deferred: a per-user "TTS delay multiplier" global setting (so the streamer can globally slow down or speed up TTS without editing each template), per-mapping override (different delay for the same template depending on which event triggered it), and a "test play this expression with the delay" button in the editor. None of those are worth building before we see whether the per-template delay is actually the right level of control.

## May 12th, 2026 - TTS gate accepts a `tts` boolean control on any of the user's overlays, not just user-scoped

- The first cut of the TTS gate (earlier today) only matched `OverlayControl` rows with `overlay_template_id IS NULL` (user-scoped). But the UI hint I shipped told streamers to add the boolean from the Controls tab on a static overlay, which creates a *template-scoped* control - so toggling it never gated TTS. Caught by the first user to actually try it.
- Loosened `AlertExpressionRenderer::isGatedOff()` to a single existence query: `OverlayControl::where(user_id, $user->id)->where('key', 'tts')->where('type', 'boolean')->where('value', '0')->exists()`. Any boolean `tts` control owned by the user that's set to off mutes TTS - it doesn't matter which overlay it lives on. Mental model lines up with what streamers expect: one switch, anywhere, mutes TTS.
- Updated the TTS-tab explainer copy in `templates/edit.vue` to match: "Add a boolean control with the key `tts` to any of your overlays. When the control is off, TTS is skipped for *all* your alerts." Removed the misleading "user-scoped" jargon.
- New test in `AlertTtsExpressionTest`: a `tts=0` boolean control with a non-null `overlay_template_id` (i.e. attached to a specific overlay) now also gates correctly. Existing user-scoped test still passes - same path, different `overlay_template_id` value.

## May 12th, 2026 - TTS Expressions: per-alert-template spoken-line composer, gated by a reserved `tts` boolean control

- New "TTS Expression" field on every alert template - a short templated string the server renders at dispatch time and ships in the `AlertTriggered` broadcast payload as `tts_text`. The overlay calls `SpeechSynthesisUtterance` with that string when an alert arrives, so streamers can finally have their resub / cheer / donation alerts speak a tailored line out loud without bolting on a third-party TTS service. Originally sketched as a "TTS Control" in the issue thread, but the right shape turned out to be: rendered string in the payload (so it can also be reused as plain text by any consumer), authored with the same Bot-Expressions-shaped editor, gated by a single boolean.
- New `tts_expression` text column on `overlay_templates` (nullable, only meaningful when `type='alert'`). Validated on `OverlayTemplateController::store/update` as `nullable|string|max:2000`. The HtmlSanitizationService allowlist deliberately does *not* include this field - tags inside `[[[...]]]` are inputs to the resolver, never raw HTML, and the renderer's single-pass guarantee keeps user content from being interpreted as template syntax.
- New `App\Services\Expressions\AlertExpressionRenderer`. Same single-pass regex (`/\[\[\[([\w.:\-]+)(?:\|([\w.:\- ]+))?]]]/`) as `BotExpressionResolver`, same "tags never reparse" day-one rule, but the lookup table is the flat `$templateData` payload that's already built for the alert. `event.user_name`, `event.streak_months`, bare Twitch tags like `user_name`, and `c:key` / `c:service:key` control references all work identically to how they resolve in the alert HTML. Output is capped at 500 characters; empty/whitespace-only results render as `null` so the broadcast doesn't ship junk.
- Pipe-formatter logic was duplicated between the resolver-to-build and the resolver-just-built, so I extracted it once into `App\Services\Expressions\ExpressionFormatter::apply(string $value, string $pipe, string $locale)`. `BotExpressionResolver` now delegates; `AlertExpressionRenderer` uses the same helper. All 8 formatters (`round`, `number`, `currency`, `date`, `uppercase`, `lowercase`, `distance`, `duration`) are shared. The 24 existing bot-expression tests are still green, so the refactor is transparent to existing callers.
- `AlertTriggered` event gained a `?string $ttsText` constructor param and a `tts_text` key in `broadcastWith()`. All three dispatch sites pass it through after rendering: `TwitchEventSubController::renderEventAlert`, `ExternalAlertService::dispatch`, and `ExternalEventController::replay`. The constructor parameter is optional (defaults to `null`), so any call site that doesn't care - tests, future internal callers - keeps working without changes.
- Gating uses a reserved key convention: if the user owns a user-scoped (overlay_template_id null) boolean control with key `tts` and its value is "0", the renderer short-circuits and returns `null`. Absence of the control means TTS is on - the default for any new user. There's no special tag the streamer needs to put in the alert HTML; one switch in the controls panel mutes all of their alerts globally. Documented in the editor's TTS tab so the muting workflow is discoverable.
- Editor surface lives on `templates/edit.vue` as a new "TTS" tab (alert-only, Volume2 icon, 9th tab after Triggers/Targeting). Multi-line textarea bound to `form.tts_expression` (max 2000 chars), inline hint listing the existing extracted `template_tags` as clickable chips (click to append `[[[tag]]]` to the expression), and a small explainer card on how to mute via the `tts` control. No live preview pane for v1 - the expression shares the same tag set as the alert HTML so authors can iterate by editing the HTML body and replaying a stored event.
- Overlay-side wiring in `OverlayRenderer.vue::handleAlertTriggered` is intentionally tiny: read `alertData.tts_text`, bail if not a non-empty string or `window.speechSynthesis` is missing, otherwise cancel any in-flight utterance and `speak(new SpeechSynthesisUtterance(text))`. Cancel-on-new matters because back-to-back alerts (raid + sub-train) would otherwise queue a backlog of speech the streamer can't catch up with. Errors are caught and logged to console - TTS failures should never crash the overlay.
- 6 new feature tests in `tests/Feature/AlertTtsExpressionTest.php`: rendered string ships when expression is set; `tts_text` is null when expression is null; null when expression resolves to empty (e.g. unknown tag); suppressed when the `tts` boolean control is "0"; permitted when "1"; `broadcastWith()` exposes the field positionally. Tests drive the `ExternalEventController::replay` path because it's the shortest route to a real broadcast without faking webhook signing.
- Deliberately deferred: per-alert-type TTS expressions (currently one per template - if you have one alert template serving two event types via two mappings, both will say the same thing), a live preview pane on the TTS tab, a voice picker (currently uses the browser's default SpeechSynthesisUtterance voice), and conditional / if-else syntax inside the expression. All four are follow-ups gated on real usage feedback rather than guesses.

## May 10th, 2026 - Bot Expressions: server-side backend for user-authored chat commands that read from the controls layer

- Added the entire backend for Bot Expressions, the feature first sketched in [issue #104](https://github.com/jasperfrontend/overlabels/issues/104). A streamer can now define a chat command (`!distance`, `!shoutout`, etc.) whose response is a templated string that reads from `OverlayControl` values, Twitch Helix data, and per-invocation chatter context. The bot speaks the resolved string. Bot stays dumb: it does not parse tags, does not enforce permissions, does not know about controls. All resolution and gating runs server-side before the row hits `bot_chat_outbox`.
- Mental model: Bot Expressions sit on the **read** side of the controls layer, alongside Alerts and Overlay Templates. Recipes (planned) sit on the write side. The architecture diagram lives in `resources/help/reference/Bot-Expressions.md` and `resources/help/reference/Recipes.md`.
- New `bot_expressions` table: `user_id`, `command`, `permission_level`, `cooldown_seconds`, `expression`, `enabled`, `hidden_from_commands`, `last_fired_at`. Unique on `(user_id, command)`. Per-channel cooldowns only for v1, enforced via `last_fired_at` + broadcaster bypass. Per-user cooldowns are a v2 add when demand surfaces.
- Tag syntax inside `[[[...]]]` matches overlay templates exactly, plus a new `bot:` namespace for invocation context. `[[[c:goal_km]]]` for own controls, `[[[c:kofi:donations_received]]]` for service-managed controls (resolved via `broadcastKey`), bare names like `[[[followers_total]]]` for Twitch Helix data, `[[[bot:from_user]]]` / `[[[bot:args.0]]]` for chatter context. Authors copy-paste freely between overlay templates and Bot Expressions.
- `BotExpressionResolver` is the new server-side template resolver in `app/Services/Bot/`. Net-new code: until now all tag substitution lived in the frontend `tagParser.ts` / `OverlayRenderer.vue`. Same regex shape, same single-pass-by-design rule (substituted values are never re-scanned for tags), same null-over-placeholder rule. Output is hard-capped at 500 characters to stay under Twitch's chat message limit. Pipe formatters supported: `round`, `number`, `currency`, `date`, `uppercase`, `lowercase`, `distance` (m -> km/m/mi/ft), `duration` (dd:hh:mm:ss patterns). Locale comes from `User::preference('locale')`.
- `BotExpressionService` orchestrates: `canFire()` runs enabled + permission tier ladder + cooldown, `fire()` resolves and writes the outbox row, stamps `last_fired_at`. `buildBotContext()` flattens the bot's POST payload into the resolver-friendly shape including indexed `args.0`, `args.1`, ... tokens.
- New internal endpoint `POST /api/internal/bot/expressions/fire`. Bot POSTs chatter context (login, display name, badges, args), Overlabels resolves and queues. Silent-on-block policy: failures return 200 OK with `queued: false` and a reason, no chat reply, so the bot doesn't spam "you're on cooldown" noise. Sits behind the existing `bot.internal` middleware + `throttle:bot-internal` bucket - no new auth surface.
- Extended `BotCommandController::index` to merge `BotExpression` rows into the per-user command map with `type: "expression"`. Existing builtins are now tagged `type: "builtin"`. The bot uses `type` to route: hardcoded JS handler for builtins, POST to the fire endpoint for expressions. On collision (a user authored an expression with the same command as a builtin), builtin wins both at the merge layer and at validate-on-save time. JSON shape change is additive, so an old bot deploy reading the response will still work - it'll just ignore the new field.
- Permission enum reuses `BotCommand::PERMISSION_LEVELS` exactly: `everyone` / `subscriber` / `vip` / `moderator` / `broadcaster`. Tier ladder: a `vip`-badged chatter passes a `subscriber`-tier expression, a `moderator` passes a `vip`-tier, etc. Resolution uses the `badges` array the bot includes in the fire POST.
- 24 new feature tests in `tests/Feature/BotExpressionsApiTest.php` covering auth, lookup, gating (disabled / wrong permission / on cooldown), broadcaster bypass, resolution of all three tag families, single-pass safety, dry-run, length cap, sync-endpoint merge behaviour, and collision policy. One existing assertion in `BotInternalApiTest.php` updated to include the new `type` field. Full suite green: 534 passed.
- Three new reference docs in `resources/help/reference/`: `Bot-Expressions.md` (formal spec, manifest now serves as a build contract), `Recipes.md` (the producer-side primer that frames the controls-layer-as-bus architecture), and `Wheel of Fortune.md` (the first concrete Recipe / Kit candidate). These are design docs for the next chunks of work, not user-facing help.
- Not in this commit (deferred to follow-ups): the Inertia builder UI under `settings/bot/expressions` (substantial frontend chunk - own session), the bot-side dispatch branch in the separate `overlabels-bot` repo (tiny `bot.js` change to recognise `type: "expression"` and POST to fire), and the auto-generated tag-discovery reference page. The Laravel-side contract is concrete and tested, so those follow-ups can land independently.

## May 9th, 2026 - Fix: donation-seed and test-mode reset now target `total_received` instead of `donations_received`, and the duplicated logic moves to `ExternalControlService`

- The donation-goal seed across all five integration controllers (Ko-fi, BMAC, Fourthwall, StreamElements, StreamLabs) was writing the user's "starting amount" into the `donations_received` counter instead of the `total_received` running amount. A streamer entering "30" because they'd already raised €30 before connecting Overlabels was seeding "30 separate donations have happened" - which is nonsense and made downstream donation-goal templates render zero progress until the first real donation arrived. Same bug existed in the test-mode toggle: turning test mode off was resetting `donations_received` to the seed value and `total_received` to 0, exactly inverted. Both paths now write the seed to `total_received` and reset `donations_received` to 0 like every other counter.

## May 4th, 2026 - Fix: cascade overlay-controls on template delete (was throwing 500)

- Deleting a template in prod started returning 500. The partial unique index `overlay_controls_user_source_key_unique (user_id, source, key) WHERE overlay_template_id IS NULL` then collided with the user-scoped service-managed row already at `(user, source, key, template_id=NULL)` from auto-provision at integration-connect time, throwing `SQLSTATE 23505`. Diagnosed by grepping the prod web container's docker logs for the 500: stack trace pointed straight at `OverlayTemplateController::destroy` line 259 hitting the unique index inside the FK demotion.

## May 2nd, 2026 - Fix: split bot internal throttle so gamejam votes can't 429 the whole bot
