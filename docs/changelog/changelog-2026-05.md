# CHANGELOG MAY 2026

## May 13th, 2026 - Lists page: fix phantom-blank-first-item + subscribe to live broadcasts

- Two post-ship fixes after first real prod use surfaced both. (1) Creating an empty list then having a chat appender add to it produced a list with a phantom blank first row. (2) The /dashboard/lists page didn't update when a chat command appended to a list - had to reload the page to see the new item.
- Phantom blank fix in `resources/js/pages/dashboard/lists/index.vue`: the textarea was sending `newItemsText.value.split('\n')` to the server on save. JS's split returns `[""]` (one empty-string item) when called on an empty string, not `[]` - so saving an empty textarea created a list whose first item was already `""`, and `!hi hello` from the first chatter would append AFTER that phantom. Fix is `text === '' ? [] : text.split('\n')`. Same fix on `createList()` and `saveActive()`. Preserves the "lists are lists, don't touch what was typed" contract for any non-empty content. Matches what Streamer.Bot's Write-To-File feature is doing under the hood too - same underlying JS pitfall.
- Live updates fix: the /dashboard/lists page wasn't subscribed to the user's broadcast channel at all. Only `OverlayRenderer.vue` was. Added an Echo subscription in onMounted using the same `alerts.{twitch_id}` pattern as `useStreamState`. Listens for `.list.updated` and `.list.deleted`; patches the local `lists` ref by slug. If the user is currently editing the affected list AND has unsaved changes, the active textarea is left alone (their pending edits win until they save); if they're not dirty, the textarea refreshes with the new items so a chatter's append appears in the view immediately. onUnmounted leaves the channel cleanly.
- No backend changes - the broadcast was already firing from `ListAppendService`. Both fixes are entirely in the Vue page.

## May 13th, 2026 - List Appenders: chat commands that grow Lists (raffle / queue / quotes / praise wall)

- Builds the write side on top of the Lists feature shipped earlier today. Streamers can now wire a chat command like `!raffle` to append the invoker's display name to a List automatically; same shape extends to `!join` queues, `!quote` walls, etc. The read side (Bot Expressions / overlays reading lists) and the write side (chat appending to lists) compose to close the producer/bus/consumer loop for the user-controlled-data path.
- Two new tables. `list_appenders` (user_id, target_list_id FK to option_sets, command, permission_level, cooldown_seconds, value_template, args_empty_reply, dedup_policy, max_size, enabled, last_fired_at, timestamps; unique(user_id, command)). `list_append_history` (list_appender_id FK, target_list_id FK, chatter_id, chatter_login, value, stream_session_id FK nullable, fired_at; append-only, no updated_at). The history table powers dedup + provides an audit trail without requiring a join on every check.
- The value template uses the exact same syntax as Bot Expressions - same `[[[bot:from_user]]]` / `[[[bot:args]]]` / `[[[c:foo:bar]]]` / pipe formatters - because ListAppendService just hands the template to the existing `BotExpressionResolver`. Zero new template language. The same applies to `args_empty_reply` - "Hi @[[[bot:from_user]]] add something after !raffle, like !raffle salami!" resolves chatter context via the same code path.
- `[[[bot:fired_at]]]` added to the shared bot context in `BotExpressionService::buildBotContext` - Unix seconds at dispatch time. Available to both Bot Expressions and List Appenders now. Pipe formatters give it locale-aware formatting for free: `[[[bot:fired_at|date:HH:mm]]]`.
- Three dedup policies: `none` (allow duplicates - quotes, praise walls), `per_chatter` (lifetime per appender - raffles), `per_chatter_per_stream` (resets when stream goes live - game queues, where each viewer should get one play per stream). When stream is offline and `per_chatter_per_stream` is configured, falls back to lifetime `per_chatter` to keep offline tests from accidentally seeding the next stream's queue.
- `max_size` is enforced with a `lockForUpdate` inside the fire transaction so two simultaneous chat fires at slot 99/100 can't both pass the check and over-fill the list. Silent refuse on full (no chat message) - streamer can layer a Bot Expression that reads `:count` for a "raffle is full" announcement if they want one.
- Args-empty handling: when the value template references `[[[bot:args*` AND the chatter supplied no args, the fire bounces with `reason: 'args_empty'`. If the streamer configured an `args_empty_reply` (Bot Expression syntax), it's resolved with chatter context and queued into `bot_chat_outbox` - the bot's existing outbox poller picks it up and speaks it. Bot stays dumb. Leaving `args_empty_reply` null gives silent refusal for streamers who want zero chat noise.
- `BotExpressionResolver` reuse made the service shockingly thin: ~150 lines for canFire + fire + dedup + history-write + broadcast. Same shape as `BotExpressionService` and `RecipeChatTriggerService` so the three are now interchangeable in pattern, just different terminal effects (output to chat / fire a picker / append to a list).
- Internal API: `POST /api/internal/bot/list-appenders/fire`, X-Internal-Secret auth, same payload shape as `/expressions/fire` and `/recipe-triggers/fire`. `BotCommandController::index` now surfaces appenders with `type: 'list_append'` in the commandMap. Resolution order on collision: builtin > expression > recipe_trigger > list_append. ListAppenderController (web side) refuses to save a command that collides with any of the other three surfaces.
- Bot repo `6e647d0..` adds the `entry.type === 'list_append'` branch to `bot.js`, calling a new `fireListAppender(payload)` in `overlabelsApi.js`. Mirrors the recipe_trigger dispatch path exactly. Bot remains silent in chat for the success path; args_empty_reply paths reach the outbox via Overlabels-side code and get spoken by the existing outbox poller.
- UI: per-list "Append commands" section on `/dashboard/lists` (right pane, under the items textarea). One row per appender showing command, permission, dedup, max_size, cooldown, value template, args_empty_reply preview. Add/Edit via Reka Dialog modal with all fields. CRUD goes through `ListAppenderController` returning JSON (consumed by axios) rather than Inertia full-page reloads - feels inline instead of router-bouncy.
- 14 new Pest feature tests in `ListAppenderTest`: append + history write + ListUpdated broadcast, per_chatter dedup, none dedup allows duplicates, max_size silent refuse, args_empty queues outbox reply, args_empty silent when no reply configured, multi-tag value template resolution, fire endpoint happy/channel_not_found/appender_not_found/gate paths, X-Internal-Secret enforcement, commandMap surface + collision resolution. 637 tests green (was 623).
- Deliberately deferred: EventSub-triggered appenders ("every new subscriber goes into a recent_subs list" - the table is generalisable but the source field isn't there yet), other list ops (remove/clear/replace - currently append-only), streamer-facing history view ("who joined the raffle and when" - data's already there in `list_append_history`, no UI yet), and visible "raffle is full" / "you're already in" replies from the appender itself (use Bot Expressions for those).

## May 13th, 2026 - Lists: user-managed OptionSets as a first-class top-level feature

- After reverting the Wheel kit's bad frontend implementation earlier today (commits 7ef9457, d009aec, 1317e39), the actionable insight was: Recipes today don't have a backend rich enough to think about visuals. OptionSets only existed when a recipe install created them - users couldn't author their own. This commit promotes them to first-class. Recipes are still the producer-of-controls layer; what we did is uncouple "lists of values" from "recipes that pick from lists." Lists are a primitive that anyone can create and any consumer can read; recipes are one such consumer.
- The `option_sets` table didn't need a schema change (recipe_instance_id was already nullable). New surface is purely additive: routes, controller, Vue page, broadcast event, renderer injection.
- `ListController` at `app/Http/Controllers/ListController.php` handles index / store / update / destroy under `/dashboard/lists`. User-authored lists (recipe_instance_id = NULL) get full CRUD. Recipe-installed lists also surface here and are editable when their recipe declared `user_editable: true`; locked ones stay visible but are read-only via a 403 on the update path. Min/max items from a recipe manifest are still respected on save when set.
- "Lists are lists" - server-side validation is intentionally minimal. The controller does NOT dedupe, does NOT strip empty lines, does NOT trim whitespace. Only filter applied is NUL-byte stripping per item, purely so the json_encode pipeline can't blow up on a NUL that would never be deliberate input. Empty lines, duplicates, leading/trailing whitespace, and Unicode are preserved exactly. Two existing global middlewares would have silently corrupted this contract - `TrimStrings` and `ConvertEmptyStringsToNull` - so `bootstrap/app.php` now exempts both for requests under `/dashboard/lists*` via callback closures (the Laravel 11+ API).
- New tag namespace `[[[c:list:<slug>]]]` exposed at render time. `OverlayTemplateController::renderAuthenticated` injects each user-owned list as TWO data shapes simultaneously: (1) the full JSON-encoded array under `c:list:<slug>` so direct tag substitution resolves to a JSON string, AND (2) flat indexed scalars `c:list:<slug>.0`, `c:list:<slug>.1`, ... plus `c:list:<slug>.count` so the existing `materialiseIterable` (`useConditionalTemplates.ts:176`) in the foreach machinery picks them up unchanged. Net result: `[[[foreach:c:list:donors as donor]]][[[donor]]][[[endforeach]]]` Just Works against any user list, alongside the existing foreach support for Twitch event payloads, with no parser or materialiser changes.
- Live propagation via new `ListUpdated` broadcast event on the existing `alerts.{twitch_id}` channel. Fires on save with the full items array as payload + `updated_at` (Unix seconds); fires with `items: null` on delete and uses `broadcastAs() = 'list.deleted'` to let the listener distinguish without inspecting payload shape. `OverlayRenderer.vue` registers `.list.updated` and `.list.deleted` listeners alongside the existing `.control.updated`; the update handler patches both the full JSON and the indexed scalars in-place and prunes any tail keys from a shrinking list so stale `.3`, `.4`, ... don't linger in the data store. No overlay reload needed when a streamer edits a list - the next foreach iteration on the next morphdom tick reflects the change.
- Vue page at `resources/js/pages/dashboard/lists/index.vue` is a two-pane layout: left rail lists the user's lists (label, slug, item count, recipe badge if any, lock icon if recipe-locked), right pane edits the active list (label input + raw textarea + save/delete). Textarea splits on `\n` at save time so the user can type empties and duplicates intentionally. Create flow is a collapsible card at the top with slug validation matching the server pattern (`^[a-z][a-z0-9_]{0,49}$`), live-previewed as `[[[c:list:your_slug]]]`. Snippet copy button on the active list puts the tag on the clipboard.
- AppSidebar gains a "Lists" entry under "My stuff" (between Alerts and Kits) using the lucide `ListIcon`. Lists are a primitive recipes happen to use, not a sub-feature of recipes - top-level nav reflects that.
- 15 new Pest tests in `ListControllerTest`: index returns the right shape with `[[[c:list:<slug>]]]` tag baked in, index doesn't leak other users' lists, store preserves empties and duplicates exactly, store strips NUL bytes only, store rejects malformed slugs and duplicate-per-user slugs, update wholesale-replaces items and broadcasts, update refuses locked recipe lists, update allows editable recipe lists, update enforces min/max bounds, update returns 404 on foreign lists, destroy refuses recipe-installed lists, destroy broadcasts a `list.deleted` event, and broadcast payload shape covers both update and delete cases. 623 tests green (was 608).
- Deliberately not in this slice: dynamic lists populated from Twitch/EventSub data (the planned step 3), transformations on lists (filter/sort/slice - step 4), reorder beyond "edit the textarea and resave", per-item metadata (color/weight/etc.). The static-list backbone shipping first means none of those need to fight schema choices for their UI surface later.

## May 13th, 2026 - Recipes step 2: manifest JSON Schema + PHP validator + canonical Coin Flip manifest

- Step 2 of the Recipes build order is "Define the Recipe manifest shape. JSON schema for which primitives, how wired, what controls are exported. Version field from day one. Hand-authored, no UI." This lands the formal contract that step 3's RecipeInstaller will read from and that future third-party recipe authors will publish against.
- The doc is explicit that "once recipes are shared, the manifest shape becomes a contract you cannot break" - so the manifest was authored as a real JSON Schema artifact (resources/recipes/recipe-manifest.schema.json, draft-2020-12), not Laravel-style validation rules in PHP. A standalone schema is the right shape for the Claude-auditable-before-publishing story the doc calls for: any third-party can run the same schema against their manifest before publishing, with no Overlabels code path involved.
- Composer dep: opis/json-schema ^2.6. Well-maintained, supports draft-07/2019-09/2020-12, no transitive baggage. Picked over justinrainbow/json-schema (older, draft-04 era) and league/json-schema (less complete).
- Schema encodes the locked decisions from the doc: integer recipe_format_version (const 1 for now; bumps create new installable artifacts, never in-place upgrades), snake_case slug pattern matching the existing control-namespace rules, sectioned primitives object (option_sets + pickers arrays), permission enum mirroring BotCommand::PERMISSION_LEVELS exactly (everyone/subscriber/vip/moderator/broadcaster - 'broadcaster' covers the streamer_only case so we keep one vocabulary across Bot Expressions and Recipe triggers), control_exports as {name, from} with from constrained to pickers.<ref>.{result|result_at|running} (the three picker fields actually supported today; locked by regex so a manifest trying to export a made-up internal field fails at validate-time), and triggers as oneOf chat_command/dashboard_button (EventSub and scheduled deliberately deferred per the doc). Every object uses additionalProperties: false so accidental typos in field names are caught at validate-time rather than silently ignored.

## May 13th, 2026 - Production PHP upload limits aligned with the Cloudinary endpoint's 10MB validation

- User reported a hard-to-debug 422 on `/kits/create` when pasting a screenshot. Network response was the generic Laravel `{"message":"The image failed to upload.","errors":{"image":["The image failed to upload."]}}`. The wording is the giveaway: that's the implicit `uploaded` rule message, not the `image` rule message. PHP rejected the upload at the SAPI level before Laravel's validator ever ran. Pasted PNGs from a high-res display land in the 3-8MB range, which sails past PHP's stock 2MB ceiling.
- Root cause was the `dunglas/frankenphp:1-php8.4` runtime inheriting the official PHP image's `php.ini-production` defaults (`upload_max_filesize=2M`, `post_max_size=8M`). The Dockerfile copied a Caddyfile and an entrypoint but never wrote any PHP ini overrides, so the prod container ran with limits a third of what the Laravel-side validator expected (`max:10240` = 10MB on `CloudinaryUploadController`). Symptom only showed up under specific conditions (high-DPI clipboard screenshot vs a small drag-dropped JPG), which is why it survived the original Cloudinary backend-routing rollout.
- Fix: new `docker/php-uploads.ini` snippet (`upload_max_filesize=10M`, `post_max_size=12M`) copied into `$PHP_INI_DIR/conf.d/zz-uploads.ini` in the runtime stage. `post_max_size` sits 2MB above the file cap to absorb the multipart envelope and the `kind` form field, so PHP accepts the full 10MB image and Laravel renders any further error (dimensions, mime, abuse) instead of the opaque uploaded-rule message. The `zz-` filename prefix ensures the override loads after anything else in `conf.d/` no matter how PHP orders its scan.
- Frontend tweak in `resources/js/components/ImageDropZone.vue`: error-message fallback chain reordered so `data.errors.image[0]` / `data.errors.kind[0]` are checked before `data.message`. Laravel's 422 envelope always includes the generic "The given data was invalid." (or here, the upload-rule string) as `message`, but the field-specific entry under `errors` is what the user actually needs to see. Before the swap the dropzone surfaced the generic; now it surfaces "Image must be at least 400x400px" or similar verbatim.
- No frontend code changes to `kits/create.vue` or `kits/edit.vue` were needed. Earlier suspicion that they had drifted from `templates/edit.vue`'s upload flow turned out to be a red herring - all three call the shared `ImageDropZone` with the right `kind` prop and have done since the f9a8479 backend-routing refactor.

## May 12th, 2026 - Per-alert TTS playback delay so the alert sound can play before the voice

- The first cut of TTS spoke the rendered string the moment the broadcast arrived, which collided with the alert sound/animation every streamer's templates already include. Standard alert design is "noise first, voice second" - and as long as we don't give people a way to push the voice back, every alert is going to sound like two things shouting at once. Fix: a per-alert-template delay value, configurable on the same TTS tab that already holds the expression.
- New `tts_delay_ms` column on `overlay_templates` (unsigned int, default 0, capped at 60000 by the controller validator). Sits next to `tts_expression` because they're authored together and conceptually inseparable - a delay without an expression is meaningless, and an expression without a delay is just the v1 behaviour we already had. Cast to integer on the model so `null` from the DB never sneaks into the JS layer as a string.
- `AlertTriggered` gained an `int $ttsDelayMs = 0` constructor param (clamped to `max(0, value)` in the constructor so a freak negative can never reach the wire) and a `tts_delay_ms` key in `broadcastWith()`. All 3 dispatch sites - `TwitchEventSubController::renderEventAlert`, `ExternalAlertService::dispatch`, `ExternalEventController::replay` - pass `$template->tts_delay_ms ?? 0`. The constructor param defaults to 0 so existing test fixtures that don't care about it keep compiling.
- Overlay side: `OverlayRenderer::speakTts(text, delayMs)` now schedules speech via `setTimeout` when `delayMs > 0`. The pending timer is held in a module-scoped `ttsPendingTimer` ref so a new alert arriving during the delay window cancels the previous alert's not-yet-spoken voice - otherwise a fast double-fire would queue up two delayed `speak()` calls with no way to interrupt the first. `clearTimeout` on the pending timer + `speechSynthesis.cancel()` runs unconditionally before scheduling, which collapses cancel-pending + cancel-in-flight + schedule-new into one path.
- Editor UI: a number input "Wait before speaking (ms)" sits below the TTS Expression textarea on the same tab. Step is 100, min 0, max 60000, body copy explains the "match the length of your alert sound" intent. Form binding uses `v-model.number` so the value reaches the backend as an integer, not a string.
- 4 new feature tests: ships the value through `ExternalEventController::replay`, defaults to 0 when the column is absent, exposes the field through `broadcastWith()`, and the negative-input clamp. 545 tests green (was 540).
- Deliberately deferred: a per-user "TTS delay multiplier" global setting (so the streamer can globally slow down or speed up TTS without editing each template), per-mapping override (different delay for the same template depending on which event triggered it), and a "test play this expression with the delay" button in the editor. ## May 4th, 2026 - Fix: cascade overlay-controls on template delete (was throwing 500)

- Deleting a template in prod started returning 500. The partial unique index `overlay_controls_user_source_key_unique (user_id, source, key) WHERE overlay_template_id IS NULL` then collided with the user-scoped service-managed row already at `(user, source, key, template_id=NULL)` from auto-provision at integration-connect time, throwing `SQLSTATE 23505`. ## May 2nd, 2026 - Fix: split bot internal throttle so gamejam votes can't 429 the whole bot

- 429 events are recorded into a small `app/Services/Bot/RateLimitLog` cache list (last 50 events, 24h TTL, no DB) and surfaced on `/gamejam/admin` as a "Recent rate-limited (429) requests" section above the debug toggle. Each row shows timestamp + scope + channel login, so the next time a stream throttles you can see it immediately and raise the cap. Chat-side response is intentionally unchanged: `something went wrong` stays in chat, all the diagnostic detail lives in the admin panel.
