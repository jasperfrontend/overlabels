# CHANGELOG MAY 2026

## May 4th, 2026 - Fix: GPS speed control now stores raw m/s (templates format with the speed pipe)

- Symptom: setting the integration `speed_unit` to mph and using `[[[c:overlabels-mobile:gps_speed|speed:mph]]]` produced wildly wrong numbers, while a bare `[[[c:overlabels-mobile:gps_speed]]]` looked fine. Three layers were each trying to own the unit conversion: the driver pre-converted m/s -> km/h on ingest, `beforeControlUpdates()` re-converted km/h -> mph if the integration setting demanded it, and the pipe formatter then ran a third conversion against an already-converted stored value. The setting and the pipe were fighting each other.
- The `gps_session_max_speed` and `gps_session_avg_speed` controls already followed the right pattern - store raw m/s, format at template-time. `gps_speed` was the legacy holdout, predating the pipe formatter system. Brought it in line: both `OverlabelsMobileServiceDriver` and `GpsLoggerServiceDriver` now store the inbound `speed` value as raw m/s (rounded to 4 decimals) with no conversion. Stripped the `* 3.6` from `getControlUpdates()` and the `kmh / 1.609344` mph re-conversion from `beforeControlUpdates()` on both drivers.
- `external_integrations.settings.speed_unit` is now a display hint for the GPS Sessions dashboard only - it does not influence any stored control value. Updated the integration settings copy on both `gpslogger.vue` and `overlabels-mobile.vue` to spell out that overlay templates pick their unit per tag with `|speed:kmh` or `|speed:mph`. Auto-provisioned label updated to `GPS Speed (m/s)` to match the session-aggregate label convention.
- Breaking change for existing templates: a bare `[[[c:overlabels-mobile:gps_speed]]]` (or `gpslogger`) now renders raw m/s instead of the user's preferred unit. Bit the bullet because Overlabels' first user is the only person currently using these controls and we're on his template fixing it directly. Updated the contract doc at `docs/gpslogger-backend-contract.md` to match. Tests in `OverlabelsMobileWebhookTest` and `GpsLoggerWebhookTest` updated; the old "converts speed to mph when speed_unit is mph" tests now assert `speed_unit` does NOT affect stored value. Side fix: stale control-count assertion in the mobile connect test (was hardcoded to 8, driver auto-provisions 13) replaced with a reflective `count(...->getAutoProvisionedControls())`.

## May 4th, 2026 - Fix: cascade overlay-controls on template delete (was throwing 500)

- Deleting a template in prod started returning 500. Root cause: `overlay_controls.overlay_template_id` was `nullOnDelete`, so a delete tried to demote the template's controls to `template_id=NULL` instead of removing them. The partial unique index `overlay_controls_user_source_key_unique (user_id, source, key) WHERE overlay_template_id IS NULL` then collided with the user-scoped service-managed row already at `(user, source, key, template_id=NULL)` from auto-provision at integration-connect time, throwing `SQLSTATE 23505`. The whole transaction rolled back so the user saw a delete that "didn't work" with no client-visible reason - error log only had `"message": "Server Error"`.
- Pre-existing latent bug, not caused by today's fork-wizard fix - but the fork fix would have made it more frequent, since forking a template that uses StreamElements presets now actually creates the per-template service-managed rows that hit this collision. Diagnosed by grepping the prod web container's docker logs for the 500: stack trace pointed straight at `OverlayTemplateController::destroy` line 259 hitting the unique index inside the FK demotion.
- Fix: switch the FK to `cascadeOnDelete`. When you delete a template, all its controls (plain or per-template service-managed) go with it. User-scoped rows (`template_id=NULL` from auto-provision) are unaffected because the FK only cascades on rows whose `overlay_template_id` matches the deleted row. Migration also sweeps any pre-existing orphan rows (`template_id=NULL AND source=NULL`) since they're invisible to every UI - `$template->controls()` only loads by `template_id`, ControlsManager and the service group views only see what comes through that. There's no meaningful surface where a template-less, service-less row was ever readable, so they were just dead data from prior `nullOnDelete` runs.
- Prod state before migration: 136 orphan rows, all belonging to a single user, all `source_managed=false`, types ranging across text/number/expression/etc. - all residue from past template deletions that succeeded before the user added any service-managed presets to those templates. Prod state after: 0 orphans, FK confirmed CASCADE via `pg_constraint.confdeltype = 'c'`. Migration ran via the prod web container with `php artisan migrate --force` after scp'ing the file in; the file is committed to git so the next Kamal deploy is a no-op for this migration.

## May 4th, 2026 - Fix: route non-AJAX fork entry points through the import wizard

- Companion fix to today's earlier wizard-source bug. The wizard only opened on the one fork path that uses `axios.post` (the dropdown on a template's show/edit page). The other four entry points - the Copy button on the public overlay preview at `/overlay/{slug}/public`, plus the Copy/Fork dropdown items in `TemplateCard.vue`, `TemplateList.vue`, and `TemplateTable.vue` - all submitted as either a plain HTML form or `router.post`, neither of which sets `Accept: application/json`. The controller's `wantsJson()` check went false, the redirect-to-templates-index branch fired, and the user landed on a forked template with zero controls and no chance to import them. The wizard's whole job was being skipped on most of the entry points it was built for.
- Could have rewritten each consumer to use `axios.post` and emit upward, but that's four call sites plus a blade template with no Vue context, and the wizard component lives on the show page anyway. Cleaner to make the controller redirect to `templates.show` (instead of `templates.index`) on non-JSON requests and flash the wizard payload onto the session. The show page picks it up through Inertia's shared `flash` prop and opens the wizard from `onMounted`. One server-side change, one new flash key, one onMounted hook - the four consumers don't need to know anything changed.
- `OverlayTemplateController::fork()` now builds the wizard payload (`source_controls`, `has_controls`, `required_services`, `connected_services`, `template`) once and either returns it as JSON or attaches it via `->with('fork_wizard', $payload)` on the redirect. `HandleInertiaRequests::share()` exposes `flash.fork_wizard`. `useTemplateActions` gets a new `openWizardFromPayload(data)` helper - the existing AJAX path now calls it too, so both code paths converge on the same wizard-state setter. The show page reads the flash on mount and calls the helper if present.
- Net result: clicking Copy on a public overlay preview, on a template card, in a list row, or in a table dropdown now all land you on the new template's show page with the import wizard already open. The dropdown path on a template you already own still uses the JSON path, so its UX is unchanged. Pre-existing test failures (`OverlabelsMobileWebhookTest`, `ApplyActionTest`, `ExpressionControlTest`) confirmed unrelated by stashing this commit and re-running.

- Copying a public overlay through the fork wizard was silently downgrading every service-managed control (Ko-fi, StreamLabs, StreamElements, Fourthwall, BMAC presets) into a plain template control. The wizard correctly told the user "this template uses StreamElements - you already have it connected", but on Import Selected the new copy ended up with `donations_received`, `latest_donor_name`, etc. as orphan controls with no source, no "Managed by StreamElements" badge, and no membership in the StreamElements group in ControlsManager. The user could then add a fresh `donations_received` preset on top of the orphan because the dedup checks `(template_id, key, source)` and the imported row's source was null.
- Three places in the round-trip dropped `source` and `source_managed`. `OverlayTemplate::fork()` selected only six columns into `_sourceControls`, so the wizard payload never carried them. `ForkImportWizard.vue` mapped only those same fields into wizard rows and back out to `/controls/import`. `OverlayControlController::importForkedControls()` had no validation rules for the two columns and `OverlayControl::createForTemplate()` didn't accept them either, so even if the frontend had sent them they would have been silently discarded.
- Threaded both columns end-to-end. `fork()` now selects `description`, `source`, and `source_managed` alongside the original six. The wizard's `SourceControl` and `WizardRow` interfaces gained the two fields and the watch that initialises `rows` carries them through verbatim. The submit map sends them back. Validation rules added on the controller (`nullable|string|max:50`, `nullable|boolean`). `createForTemplate()` derives `source_managed` from whether `source` is set, defaulting to `true` when present so a malformed payload cannot create a `source != null, source_managed = false` row.
- UX touch-up on the wizard table: added a Source column showing the service badge for service-managed presets, and the Key input is locked when `row.source` is set (preset keys are part of the service contract; renaming `donations_received` to `donations_received_2` would orphan it from the broadcast pipeline). Description now also rides through, since `createForTemplate()` already supported it but the wizard was throwing it away.
- Net result: forking a template that uses StreamElements now produces a copy where the StreamElements controls keep their group, their "Managed by" badge, and their broadcast key (`c:streamelements:donations_received`). The duplicate-add path is closed because dedup matches on `(template_id, key, source)` exactly as it did before. Bug 2 - the public-render Copy button bypassing the wizard entirely - is a separate fix, coming next.

## May 4th, 2026 - Feature: extend expression engine with sqrt and full inverse-trig surface

- Triggered by a friend cycling toward TwitchCon: his overlay needs a character to move across screen proportional to the great-circle distance from his live GPS to the destination. Haversine is the standard formula, and every part of it was already expressible as a chain of Controls (arithmetic, `sin`, `cos`, `PI`) except the final `R * 2 * atan2(sqrt(a), sqrt(1-a))` step. `sqrt` and `atan2` were the only two missing pieces.
- Added `sqrt`, `tan`, `asin`, `acos`, `atan`, and `atan2` to the jsep evaluator's `FUNCTIONS` map and `SUPPORTED_FUNCTIONS` set in `useExpressionEngine.ts`. The proposal originally framed this as adding `Math.sqrt`/`Math.atan2`, but Overlabels expressions use bare names (`sin(x)`, not `Math.sin(x)`) - and `abs`/`round`/`floor`/`ceil` from the proposal's "confirm" column were already there. Real gap was just the trig + sqrt surface.
- `sqrt(x)` returns 0 for negative input rather than NaN, matching the engine's existing "swallow div-by-zero, return 0" convention so a malformed control upstream can't propagate NaN through every dependent expression. `atan2(y, x)` is a two-arg call - the resolver already handled multi-arg functions (`max`, `clamp`, `mod`), so no resolver changes needed.
- Help docs updated: ExpressionBuilder modal pill row now lists the full trig set with a one-line note on what each does, the Math.vue cheatsheet table gains rows for `tan`/`asin`/`acos`/`atan`/`atan2`/`sqrt`, and the "No exponentiation operator" amber callout no longer claims `sqrt` and `tan` are unavailable - just `log` and `exp` (which remain out of scope).
- Backend stays untouched: expressions only evaluate in the frontend (`useExpressionEngine.ts`), backend just stores the string and tracks dependencies for the cascade. Verified existing Pest tests in `ExpressionControlTest.php` still pass; one pre-existing failure on main ("expression with no references should 422") was confirmed not introduced by this change.
- Net result: the full haversine formula is expressible as a portable Controls chain - destination coordinates stay in Overlabels (a streamer-configured value, not something the GPS app should own), no per-overlay JS workarounds, works for any future GPS overlay without code changes.

## May 3rd, 2026 - Security: hide Twitch ID behind a Sqids-encoded slug on the public map page

- Public map URLs were `/map/{twitch_id}`, exposing the streamer's numeric Twitch ID to anyone with the share link. The same ID also leaked through `/api/map/{twitch_id}/position`, the `.../geojson` endpoint, and the `map.{twitch_id}` Reverb channel name visible in DevTools. With a Twitch ID a chatter could URL-hack to any other registered streamer's `/map/...` URL to probe whether they were on the platform.
- Added `App\Services\MapSlugService` wrapping `sqids/sqids` (`composer require sqids/sqids` -> 0.5.0). Encodes a Twitch ID into an opaque short slug (default min length 8) using a custom alphabet pinned in `config/services.php` under `map_slug.alphabet`. Pure CPU, reversible, deterministic - zero DB or cache hits per request, which matters because a popular streamer's map page can have hundreds of viewers polling `/api/map/{slug}/position` every 5s in delay mode. Alphabet is overridable via `MAP_SLUG_ALPHABET` env var; if it ever changes, every previously shared map URL stops resolving (committed default keeps prod stable).
- All public surfaces switched to the slug:
    - Routes: `GET /map/{slug}` and `GET /map/{slug}/{sessionId}` in `web.php`; `GET /api/map/{slug}/position` and `GET /api/map/{slug}/{sessionId}/geojson` in `api.php`.
    - Controllers: `MapController::live`/`session` and `GpsSessionMapController::currentPosition`/`publicSessionGeoJson` decode the slug via `MapSlugService::decode()` and 404 on a malformed/non-resolving slug before any DB query, then look up the user by `twitch_id` exactly as before.
    - Broadcast: `MapPositionBroadcast` now carries `$slug` instead of `$twitchId` and broadcasts on `map.{slug}`. `ExternalControlService::applyUpdates` encodes the slug once per call (only when map sharing is enabled) and reuses it across the per-control loop.
    - Frontend: `window.__MAP__` ships `slug` instead of `twitchId`. `LiveMap.vue`, `SessionMap.vue`, `useMapWebSocket.ts`, and `map/app.ts` rename the prop. `OverlabelsMobileIntegrationController` builds the share URL from the encoded slug. The GPS Sessions dashboard exposes `mapSlug` instead of `twitchId` and the "Open full view" button links to `/map/${mapSlug}/${session.session_id}`.
- Net result: a chatter holding a `/map/...` URL sees only `/map/Yz7X9k2m`, the WebSocket frames carry `map.Yz7X9k2m`, and the API endpoints follow the same shape. Twitch ID never appears in any client-visible string. Slug is not security-grade (sqids are reversible), but it removes the obvious "12345 is a Twitch ID" pattern and prevents URL-hacking to other streamers by guessing.

## May 2nd, 2026 - Security: gate WebSocket fanout behind private channels and overlay-token auth

- Pentest of the overlay URL+fragment model surfaced two leaks. Anyone holding one overlay URL+token (e.g. a screen-share, a chat link, a screenshot) could derive `data.user_twitch_id` from the render response and then subscribe to the public `alerts.{user_twitch_id}` Reverb channel. That channel carried `alert.triggered`, `control.updated`, `stream.status`, and `template.updated` events in real time - including donor names, donation amounts, and live `overlabels-mobile:gps_lat`/`gps_lng` updates for streamers using map sharing. Separately, the global `twitch-events` Pusher channel was a platform-wide firehose: any unauthenticated client connected to Reverb received every Overlabels user's Twitch events.
- Channels are now private. `AlertTriggered`, `ControlValueUpdated`, `StreamStatusChanged`, `TemplateUpdated`, and `EventSubSetupCompleted` all use `PrivateChannel` instead of `Channel`. `TwitchEventReceived` now broadcasts on a per-user `private-twitch-events.{twitchId}` channel instead of the platform-wide `twitch-events`; its constructor takes a required `broadcasterId`, and `TwitchEventSubController` skips the broadcast when the broadcaster id is unknown rather than fanning out to a guessable channel. `routes/channels.php` registers auth callbacks that allow only the channel's owning streamer (logged-in dashboard) to subscribe.
- Overlays are session-less browser sources, so the standard `/broadcasting/auth` route can't sign their subscriptions. Added `POST /api/overlay/broadcasting/auth` (`OverlayBroadcastingAuthController`) that takes `{slug, token, socket_id, channel_name}`, validates the `OverlayAccessToken` via the same `findByToken` path as `/api/overlay/render` (active flag + expiry + IP allowlist all enforced), and signs the Pusher-protocol auth response only for the two channels overlays legitimately need: `private-alerts.<owner_twitch_id>` and `private-twitch-events.<owner_twitch_id>`. Anything else (different user, gamejam, presence) returns 403 before any signature is produced. Eight Pest tests cover owner success, cross-user denial, channel-allowlist denial, invalid/short/non-hex/deactivated tokens.
- `resources/js/overlay/app.js` now configures Echo with a custom `authorizer` that POSTs to the new endpoint with the URL-fragment token. `OverlayRenderer.vue`, `useStreamState.ts`, and `useEventSub.ts` switch from `Echo.channel(...)` to `Echo.private(...)`. `useEventSub` now takes a `twitchId` arg so each subscriber lands on its broadcaster's private channel.
- The public `/map/{twitchId}` page (opt-in via `map_sharing_enabled`) was the one place subscribing to alerts on purpose - to receive live GPS. Replaced that with a dedicated `MapPositionBroadcast` event on a public `map.{twitchId}` channel that ships only `gps_lat`/`gps_lng`/`gps_speed`/`gps_bearing`/`gps_tracking`. `ExternalControlService::applyUpdates` dispatches it alongside `ControlValueUpdated` only for `overlabels-mobile` controls and only when the user has map sharing enabled, so opted-in viewers still get realtime updates without leaking donations or donor names. `useMapWebSocket.ts` listens on the new public channel.
- Gamejam channels (`gamejam.{twitchId}`) stay public - the `/gamejam/live/{login}` page is intentionally viewer-facing and the snapshot is meant to be public game state.

## May 2nd, 2026 - Fix: expression controls now drive animations at monitor refresh rate

- Reported as overlay weirdness: a static template using two expression controls (`160 * sin(2 * PI * now_ms() / 5000)` and a cosine sibling) animated an image in a Lissajous orbit, but visibly chugged at ~4fps. Adding a separate random-mode control (16ms interval) and saving it a second time made the animation suddenly silky. The "save twice" part was the giveaway that two bugs were stacked.
- Root cause #1: `useExpressionEngine.ts` advanced its time-dependent heartbeat with `setInterval(..., 250)`, which capped any `now()`/`now_ms()`-driven animation at 4fps regardless of formula resolution. Swapped the heartbeat for a `requestAnimationFrame` loop so time-dependent expressions re-evaluate at the display refresh rate (typically 60Hz, with the browser auto-throttling backgrounded tabs - OBS browser sources stay foregrounded so animations keep running). The random-control "fix" was working only because the random tick mutated `data.value`'s identity every ~100ms, which retripped the expression's `watchEffect` as a side effect.
- Root cause #2: `OverlayRenderer.vue`'s initial mount destructured `random_controls` as `{ min, max }` and dropped the `interval` field, so a page refresh fell back to the 1000ms default in `startRandomTick`. Now passes `rc.interval` through.
- Root cause #3: `OverlayControlController::store()` never broadcast `ControlValueUpdated`, only `update()` did - which is why creating a random or expression control on a live overlay required a second save to take effect. `store()` now broadcasts on create for both expression (with the formula) and random (with min/max/interval) types, matching `update()`'s shape. No more "save twice to make it work."
- Net effect: expression controls using `now_ms()` now render at monitor speed without needing a sidecar random control as a frame ticker, and freshly created controls activate on the open overlay immediately.

## May 2nd, 2026 - Fix: split bot internal throttle so gamejam votes can't 429 the whole bot

- Symptom from yesterday's first chat-castle playtest: a few seconds into a busy round the backend started returning 429 to every bot request, and chat saw `something went wrong` for every command for the rest of the round. Cause: the entire `/api/internal/bot/*` route group sat behind a single `throttle:30,1` (30 req/min, keyed by IP) bucket. The bot polls `/outbox` every 2s = 30/min on its own, so the bucket was always full at idle - one extra `!join` was enough to tip it over. With 50+ players each voting per round, the bucket evaporated instantly. Laravel's default 429 body is `{"message": "Too Many Attempts."}` (no `reply` or `error` key), so the bot's `dispatchGameAction` fell through to its literal `'something went wrong'` fallback for every command.
- Fix splits the inner group into two named buckets, defined in `AppServiceProvider::boot`. `bot-internal` is 600/min by IP and covers token reads, channels, commands, controls, outbox, and settings - 20x the old budget but still bounded. `bot-gamejam-action` is 1200/min keyed per `{login}` route param so a flood in one channel can't starve another, and it carries a `Limit::response()` callback that records the 429 event before returning the JSON.
- `routes/api.php` restructured: the outer prefix group now only carries `bot.internal` middleware and the `withoutMiddleware` opt-outs; two inner `Route::middleware('throttle:bot-internal')` / `Route::middleware('throttle:bot-gamejam-action')` groups split the endpoints. Same auth/CSRF posture, just two buckets instead of one.
- 429 events are recorded into a small `app/Services/Bot/RateLimitLog` cache list (last 50 events, 24h TTL, no DB) and surfaced on `/gamejam/admin` as a "Recent rate-limited (429) requests" section above the debug toggle. Each row shows timestamp + scope + channel login, so the next time a stream throttles you can see it immediately and raise the cap. Chat-side response is intentionally unchanged: `something went wrong` stays in chat, all the diagnostic detail lives in the admin panel.
