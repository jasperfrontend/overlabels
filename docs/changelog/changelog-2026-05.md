# CHANGELOG MAY 2026

## May 3rd, 2026 - Feature: Buy Me a Coffee (BMAC) integration

- Added BMAC as the fifth donation-style integration alongside Ko-fi, StreamLabs, StreamElements, and Fourthwall. Connects via webhook with HMAC-SHA256 verification: user pastes the secret from `studio.buymeacoffee.com/webhooks/<id>` into Settings -> Integrations -> Buy Me a Coffee, Overlabels generates the webhook URL (`/api/webhooks/bmac/{token}`), user pastes the URL back into BMAC. Every event signed with `x-signature-sha256` of the raw body using the shared secret.
- Six positive event types covered: `donation.created`, `commission_order.created`, `extra_purchase.created`, `membership.started`, `recurring_donation.started`, `wishlist_payment.created`. The eight refund/cancel/update events return `{status: ignored}` 200 (per BMAC's "always 2xx or we disable the webhook" rule). All six positive events bump `donations_received` and add `data.amount` to `total_received` so a single counter tracks all BMAC support.
- `latest_donor_name`, `latest_donation_amount`, `latest_donation_message`, `latest_donation_currency`, `total_received`, plus a new `latest_support_type` (Supporter / Commission / Extra / Membership / Subscription / Wishlist) so streamers can distinguish a coffee from a commission in alert templates without a chain of `[[[if:event.type = ...]]]` blocks. Same six-key contract as Ko-fi/StreamLabs/StreamElements/Fourthwall, so swapping `c:kofi:` for `c:bmac:` (or running them through `latest()`/`oldest()`) just works.
- Privacy first: `supporter_email`, `shipping_address` (top-level and nested under `commission`), and `total_amount_charged` are stripped from the stored `raw_payload` in the driver before it reaches the controller. Email is captured into a new `private_metadata` column on `external_events` (encrypted JSON cast `encrypted:array`) plus a `supporter_email_hash` indexed sha256 column for future "unique supporters" analytics, both populated via two new optional readonly props on `NormalizedExternalEvent`. When supporters mark `note_hidden = true` (memberships and monthly support), the support_note is blanked out everywhere downstream rather than rendered as a placeholder string.
- Test mode mirrors Ko-fi: enabling appends a `_test_<microtime>` suffix to the dedup messageId so BMAC's "Send Test" button can be fired repeatedly. Disabling test mode resets all source-managed BMAC controls to defaults (with `donations_received` falling back to the seed value if set). Starting-count seeding works exactly like Ko-fi - one-time only via the BMAC settings page.
- Frontend: new `bmac.vue` settings page (clone of `kofi.vue`, retitled "Webhook secret" + 6 BMAC event-type checkboxes), `BMAC_PRESETS` exported from `controlPresets.ts`, ControlFormModal surfaces them when `connectedServices.includes('bmac')` on static templates, `SERVICE_LABELS` and the integrations index button allowlist updated. Help/Conditionals page gets a Buy Me a Coffee family with four cards (controls / common tags / payment-flavor / recurring) and its own "How It Works" panel explaining HMAC verification, the privacy posture, and shared-alert-template strategy.
- Tests: 19 unit tests in `BMACServiceDriverTest` covering all six event shapes, signature accept/reject paths, PII stripping, messageId stability, and `note_hidden` semantics across BMAC's mixed string/bool flag types. 10 feature tests in `BMACWebhookTest` covering the end-to-end webhook pipeline (HMAC, dedup, test-mode bypass, control updates, refund-event ignoring, raw-payload sanitization assertions). All 29 pass; full suite stays green except for three pre-existing failures unrelated to this change.

## May 2nd, 2026 - Security: gate WebSocket fanout behind private channels and overlay-token auth

- Pentest of the overlay URL+fragment model surfaced two leaks. Anyone holding one overlay URL+token (e.g. a screen-share, a chat link, a screenshot) could derive `data.user_twitch_id` from the render response and then subscribe to the public `alerts.{user_twitch_id}` Reverb channel. That channel carried `alert.triggered`, `control.updated`, `stream.status`, and `template.updated` events in real time - including donor names, donation amounts, and live `overlabels-mobile:gps_lat`/`gps_lng` updates for streamers using map sharing. Separately, the global `twitch-events` Pusher channel was a platform-wide firehose: any unauthenticated client connected to Reverb received every Overlabels user's Twitch events.
- Channels are now private. `AlertTriggered`, `ControlValueUpdated`, `StreamStatusChanged`, `TemplateUpdated`, and `EventSubSetupCompleted` all use `PrivateChannel` instead of `Channel`. `TwitchEventReceived` now broadcasts on a per-user `private-twitch-events.{twitchId}` channel instead of the platform-wide `twitch-events`; its constructor takes a required `broadcasterId`, and `TwitchEventSubController` skips the broadcast when the broadcaster id is unknown rather than fanning out to a guessable channel. `routes/channels.php` registers auth callbacks that allow only the channel's owning streamer (logged-in dashboard) to subscribe.
- Overlays are session-less browser sources, so the standard `/broadcasting/auth` route can't sign their subscriptions. Added `POST /api/overlay/broadcasting/auth` (`OverlayBroadcastingAuthController`) that takes `{slug, token, socket_id, channel_name}`, validates the `OverlayAccessToken` via the same `findByToken` path as `/api/overlay/render` (active flag + expiry + IP allowlist all enforced), and signs the Pusher-protocol auth response only for the two channels overlays legitimately need: `private-alerts.<owner_twitch_id>` and `private-twitch-events.<owner_twitch_id>`. Anything else (different user, gamejam, presence) returns 403 before any signature is produced. Eight Pest tests cover owner success, cross-user denial, channel-allowlist denial, invalid/short/non-hex/deactivated tokens.
- `resources/js/overlay/app.js` now configures Echo with a custom `authorizer` that POSTs to the new endpoint with the URL-fragment token. `OverlayRenderer.vue`, `useStreamState.ts`, and `useEventSub.ts` switch from `Echo.channel(...)` to `Echo.private(...)`. `useEventSub` now takes a `twitchId` arg so each subscriber lands on its broadcaster's private channel.
- The public `/map/{twitchId}` page (opt-in via `map_sharing_enabled`) was the one place subscribing to alerts on purpose - to receive live GPS. Replaced that with a dedicated `MapPositionBroadcast` event on a public `map.{twitchId}` channel that ships only `gps_lat`/`gps_lng`/`gps_speed`/`gps_bearing`/`gps_tracking`. `ExternalControlService::applyUpdates` dispatches it alongside `ControlValueUpdated` only for `overlabels-mobile` controls and only when the user has map sharing enabled, so opted-in viewers still get realtime updates without leaking donations or donor names. `useMapWebSocket.ts` listens on the new public channel.
- Gamejam channels (`gamejam.{twitchId}`) stay public - the `/gamejam/live/{login}` page is intentionally viewer-facing and the snapshot is meant to be public game state.

## May 2nd, 2026 - Fix: expression controls now drive animations at monitor refresh rate

- Reported as overlay weirdness: a static template using two expression controls (`160 * sin(2 * PI * now_ms() / 5000)` and a cosine sibling) animated an image in a Lissajous orbit, but visibly chugged at ~4fps. Adding a separate random-mode control (16ms interval) and saving it a second time made the animation suddenly silky. The "save twice" part was the giveaway that two bugs were stacked.
- Root cause #1: `useExpressionEngine.ts` advanced its time-dependent heartbeat with `setInterval(..., 250)`, which capped any `now()`/`now_ms()`-driven animation at 4fps regardless of formula resolution. Swapped the heartbeat for a `requestAnimationFrame` loop so time-dependent expressions re-evaluate at the display refresh rate (typically 60Hz, with the browser auto-throttling backgrounded tabs - OBS browser sources stay foregrounded so animations keep running). The random-control "fix" was working only because the random tick mutated `data.value`'s identity every ~100ms, which retripped the expression's `watchEffect` as a side effect.
- Root cause #2: `OverlayRenderer.vue`'s initial mount destructured `random_controls` as `{ min, max }` and dropped the `interval` field, so a page refresh fell back to the 1000ms default in `startRandomTick`. Now passes `rc.interval` through.
- Root cause #3: `OverlayControlController::store()` never broadcast `ControlValueUpdated`, only `update()` did - which is why creating a random or expression control on a live overlay required a second save to take effect. `store()` now broadcasts on create for both expression (with the formula) and random (with min/max/interval) types, matching `update()`'s shape. No more "save twice to make it work."
- Net effect: expression controls using `now_ms()` now render at monitor speed without needing a sidecar random control as a frame ticker, and freshly created controls activate on the open overlay immediately.

## May 2nd, 2026 - Fix: split bot internal throttle so gamejam votes can't 429 the whole bot

- Symptom from yesterday's first chat-castle playtest: a few seconds into a busy round the backend started returning 429 to every bot request, and chat saw `something went wrong` for every command for the rest of the round. Cause: the entire `/api/internal/bot/*` route group sat behind a single `throttle:30,1` (30 req/min, keyed by IP) bucket. The bot polls `/outbox` every 2s = 30/min on its own, so the bucket was always full at idle - one extra `!join` was enough to tip it over. With 50+ players each voting per round, the bucket evaporated instantly. Laravel's default 429 body is `{"message": "Too Many Attempts."}` (no `reply` or `error` key), so the bot's `dispatchGameAction` fell through to its literal `'something went wrong'` fallback for every command.
- Fix splits the inner group into two named buckets, defined in `AppServiceProvider::boot`. `bot-internal` is 600/min by IP and covers token reads, channels, commands, controls, outbox, and settings - 20x the old budget but still bounded. `bot-gamejam-action` is 1200/min keyed per `{login}` route param so a flood in one channel can't starve another, and it carries a `Limit::response()` callback that records the 429 event before returning the JSON.
- `routes/api.php` restructured: the outer prefix group now only carries `bot.internal` middleware and the `withoutMiddleware` opt-outs; two inner `Route::middleware('throttle:bot-internal')` / `Route::middleware('throttle:bot-gamejam-action')` groups split the endpoints. Same auth/CSRF posture, just two buckets instead of one.
- 429 events are recorded into a small `app/Services/Bot/RateLimitLog` cache list (last 50 events, 24h TTL, no DB) and surfaced on `/gamejam/admin` as a "Recent rate-limited (429) requests" section above the debug toggle. Each row shows timestamp + scope + channel login, so the next time a stream throttles you can see it immediately and raise the cap. Chat-side response is intentionally unchanged: `something went wrong` stays in chat, all the diagnostic detail lives in the admin panel.
